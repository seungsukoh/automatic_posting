# 전문가 그룹 구성과 운영 방식

## 1. 기본 원칙

이 프로젝트는 일반적인 게시 도구 개발이 아니라 SNS 플랫폼 API, 광고성 메시지 정책, 개인정보, 계정 권한, 예약 발행 안정성이 함께 얽힌 프로젝트다. 따라서 개발팀만으로 진행하지 않고 정책/기술 가능성 검증 TF를 먼저 운영한다.

## 2. 1차 TF 구성

MVP 착수 전 가장 먼저 구성할 소그룹은 다음과 같다.

- PM/PO
- 플랫폼 API 전문가
- 백엔드 개발자
- 개인정보/IT 법무 전문가
- 마케팅 담당자

## 3. 1차 TF의 핵심 결정

1차 TF는 아래 질문에 먼저 답해야 한다.

- Kakao는 KakaoTalk 일반 채팅방이 아니라 KakaoTalk Channel, Business Message, Alimtalk, Friendtalk 중 무엇으로 갈 것인가?
- Instagram은 Business 또는 Creator 계정과 Meta Graph API를 사용할 수 있는가?
- Threads 공식 API로 텍스트와 이미지 게시가 가능한가?
- 이 제품은 내부 운영 도구인가, 외부 고객용 SaaS인가?
- 광고성 메시지를 발송하는가?

## 4. MVP 상주 팀

MVP 개발 단계의 현실적인 상주 팀은 다음과 같다.

- PM/PO 1명
- 플랫폼 API 전문가 1명
- 백엔드 개발자 1명
- 프론트엔드 개발자 1명
- UX/UI 디자이너 0.5명
- QA 0.5명
- DevOps/인프라 0.5명

법무, 보안, 마케팅 전문가는 상시 투입보다 주요 의사결정 시점의 자문 형태로 시작한다.

## 5. 추가 전문가

프로젝트가 SaaS 또는 유료 서비스로 확장되면 다음 인력이 필요하다.

- 보안 엔지니어
- 고객지원/운영 담당자
- 데이터/마케팅 분석가
- 결제/과금 시스템 담당자
- 개인정보보호 책임자 또는 외부 DPO 자문

## 6. 운영 리듬

- 주 1회 제품/정책 의사결정 회의
- 주 2회 개발 진행 점검
- 플랫폼 API 검증 결과는 별도 결정 로그로 관리
- 정책상 금지되는 구현 방식은 백로그에 넣지 않는다.
- 외부 API 권한 심사 일정은 개발 일정과 분리해 관리한다.
